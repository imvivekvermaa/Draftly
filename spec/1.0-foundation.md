# Spec 1.0 — Foundation (Draftly)

The **foundation** specification: the baseline product every later feature spec
builds on. It defines **what** the application does and the acceptance criteria
it is built against. See [`ARCHITECTURE.md`](../ARCHITECTURE.md) for **how** it is
built and [`README.md`](../README.md) for setup.

> Specs are numbered in the order they were introduced (`1.0` = foundation).
> New features are added as separate files (e.g. `spec/1.1-*.md`) so the full
> history of what was built, and when, stays readable.

## 1. Goal

A full-stack app where an authenticated user can generate AI content from a
prompt, save results, and revisit their history.

## 2. Roles

| Role | Capabilities |
| --- | --- |
| Visitor | Register or sign in. Cannot access the dashboard or any content API. |
| Authenticated user | Generate content, save it, and view **only their own** history. |

## 3. Functional requirements

### 3.1 Authentication
- Email + password sign up and sign in via Supabase Auth.
- Sessions persist across reloads (cookie-based).
- Unauthenticated access to `/dashboard` or the API is rejected/redirected.

### 3.2 Content generation
- Inputs: **Content Type** (blog post, social caption, email, product
  description, ad copy), **Tone** (professional, casual, friendly, persuasive,
  humorous, formal), **Topic/Keyword** (free text, 3–300 chars).
- A dynamic prompt is built from these inputs and sent to OpenAI.
- Loading state is shown while generating; failures show a clear message.
- Generation does **not** persist automatically.

### 3.3 Saving & history
- The user explicitly saves a generated result.
- A saved record stores: user, content type, tone, topic, prompt, output,
  timestamp.
- History lists the user's saved records, newest first.

### 3.4 Bonus (implemented)
- Dark / light mode with persistence and no flash on load.
- Copy-to-clipboard and export-to-`.txt` for any result or history item.
- Real authentication (Supabase Auth), not dummy.

## 4. API contract

| Method | Endpoint | Auth | Purpose |
| --- | --- | --- | --- |
| POST | `/api/generate` | Required | Generate content; returns `{ output, prompt }`. |
| POST | `/api/content` | Required | Persist a generated result; returns the saved record. |
| GET | `/api/content` | Required | Return the user's history. |
| POST | `/api/auth/account-exists` | Public | Reports whether an email is registered, so the sign-in form can tell "no account" apart from "wrong password". |

**Success:** `{ "data": <payload> }`.
**Error:** `{ "error": { "code": <ApiErrorCode>, "message": string, "fields"?: {} } }`.

Error codes: `VALIDATION_ERROR` (422), `UNAUTHORIZED` (401),
`AI_PROVIDER_ERROR` (502), `DATABASE_ERROR` (500), `INTERNAL_ERROR` (500).

## 5. Data model

- **profiles**: `id` (uuid, = auth user id), `email`, `created_at`.
- **contents**: `id`, `user_id` → profiles, `content_type`, `tone`, `topic`,
  `prompt`, `output`, `created_at`. Indexed on `(user_id, created_at desc)`.

## 6. Non-functional requirements
- Type-safe end to end (TypeScript, Zod, Drizzle, typed API client).
- Every source file under 250 LOC; single-responsibility modules.
- Input validated on the client and re-validated on the server.
- Per-user data isolation is enforced in the API layer: every content query is
  scoped to the authenticated user's id, which is taken from the verified
  session and never from client input.

## 7. Acceptance criteria
- [ ] A new user can register, sign in, and reach the dashboard.
- [ ] Generating with valid input returns content; invalid input is rejected.
- [ ] Saving adds the item to history immediately and it persists on reload.
- [ ] A user cannot read another user's history (every query is scoped to the
      authenticated user's id).
- [ ] AI/database failures surface a friendly message, never a crash.
- [ ] Dark/light, copy, and export all work.
